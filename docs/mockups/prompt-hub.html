2.1 项目定位

一句话定位

一款服务于"AI 编程手动挡阶段"的桌面应用，主形态为快捷键唤起的全屏仪表盘、辅形态为副屏常驻视图，承载提示词资产的展示、调用和沉淀，以及人机协作的协议对齐。

这不是什么

明确边界比描述功能更重要。这个项目不是：

  • 不是"更漂亮的提示词管理器"——Snippety / Paste / Promptler 已经很多，我们不和它们竞争
  • 不是"更快的提示词调用工具"——Raycast / 文本替换在调用速度上永远更快
  • 不是"AI 自动驾驶平台"——不涉及自动化执行、Agent 编排、workflow 引擎
  • 不是"面向团队协作的知识库"——单人使用，不考虑权限、评论、分享

这是什么

  • 一个手动挡阶段的练车场，帮使用者在高强度 AI 编程中建立稳定的认知节奏
  • 一个外部化的思考工具，把"我可以怎么想这件事"从头脑里卸载到屏幕上
  • 一个资产沉淀场，让每天的提示词使用自动产生可以复用、可以迭代的资产
  • 一个人机协议握手台，让每次协作开始前的"对齐动作"显式化、工具化
  • 一个双形态承载的仪表盘——主形态为快捷键唤起的全屏覆盖窗口（80% 使用场景），辅形态为副屏常驻视图（深度工作和复盘场景）

用户画像

使用者是一人公司/独立开发者，高频使用 Claude Code 进行 AI 辅助编程。 具备以下特征：

  • 每天与 AI 对话频次 > 50 次
  • 已经形成一套个人的提示词 / 工作流话术（几十条以上规模）
  • 有持续迭代和优化自己工作方法的习惯
  • 重视"看见全局"胜过"操作最快"
  • 对 AI 编程范式有自己的演进路径规划（SOP 化 → 手动 → 自动 → 协同 → 自主）

2.2 九条设计哲学

这九条哲学是整个项目的地基。任何具体设计决策都必须能回溯到这里某一条，否则这个设计就不该存在。

哲学一：五阶段演进路径

AI 编程的成熟度分五个阶段，这个项目只服务于第二阶段：

阶段特征对应工具
1. SOP 化把隐性工作方法显式化为话术、模板笔记本、文档
2. 手动挡人完整驱动，工具辅助展示和调用本项目
3. 自动挡工具自动跑一段，人设定起点和终点Claude Code slash commands
4. 辅助驾驶人机实时协同决策Claude Code plan mode + subagents
5. 自主驾驶AI 端到端交付目前不可达
为什么用"汽车驾驶"做比喻

选择汽车而不是常见的"初级/中级/高级"分级，是因为汽车比喻内嵌了两个其他分级不具备的核心前提：

前提一：驾驶员永远是决策者。哪怕是最先进的辅助驾驶，方向盘依然在人手里——这和 AI 编程的理想状态完全一致。我们不是在训练 AI 替代自己，是在训练自己和 AI 形成稳定的协同关系。"初级→高级"这种分级暗示了"终点是人不再需要干预"，但汽车比喻明确拒绝这个暗示。

前提二：每一阶段都需要前一阶段的积累。不能跳过手动挡直接开辅助驾驶——你的脚会不知道该放哪里、你的眼睛会不知道该看哪里、你的肌肉会不知道该在哪一刻紧张。能力的跃迁不是软件升级，是肌肉记忆的沉淀。

本项目在五阶段中的精确位置

本项目只做手动挡，不做相邻阶段：

  • 不做 SOP 化：话术的创作、提炼应该在 Obsidian 或笔记软件里完成。本项目不是编辑器，是展示器和调用器。
  • 不做自动挡：一键触发自动化、编排多步执行、自主跑 agentic loop——这些应该在 Claude Code 的 slash commands、composite commands、subagents 机制里实现。本项目可以为未来迁移到自动挡做准备（比如 SOP 模板可以直接映射为 composite commands），但自身不执行任何自动化动作。

哲学二：看见全局 > 操作最快

对手动挡学员来说，视觉全景比操作速度更重要。这条颠覆了效率工具的常见假设——多数效率工具追求"最少点击""最少步骤"。本项目主动接受"多一步点击"的代价，换取"一眼看全"的收益。

设计上的含义：默认视图是全景式的，不是搜索框式的；相位、场景、话术、最近使用、SOP 进度应同屏可见；拒绝把关键信息隐藏在 Tab、下拉菜单、二级页面后面。

哲学三：注意力分层通过"分离"实现，但分离方式因形态而异

仪表盘和主工作屏必须存在某种"分离"，分离才能创造注意力分层。但分离的实现方式因形态而异——主形态用时间分离，辅形态用空间分离。

认知科学上，把工具信息从主工作流卸载到一个"另外的地方"，形成 external cognition offloading（外部认知卸载）。主屏是执行现场，仪表盘是思考工具。从执行态切回决策态需要一个触发动作——这个动作不是为了换屏幕看，是为了换大脑模式。

分离方式形态触发动作退出动作
时间分离（主形态）快捷键唤起的全屏覆盖窗口按下全局快捷键（如 ⌥ Space）复制后自动隐藏 / ESC 关闭
空间分离（辅形态）副屏常驻视图抬头看副屏不需要退出（常驻在场）

为什么主形态选时间分离：不占视觉空间（主屏 100% 留给和 AI 的对话）；召之即来（任何设备都能用，不需要副屏硬件）；用完即走（复制后窗口立刻消失，主屏粘贴流程顺滑）；认知触发更强（"按下快捷键"这个动作本身就是一次明显的模式切换）。

反方案（明确拒绝的）：

  • ❌ 主屏浮层——既不时间分离也不空间分离，遮挡主屏破坏专注
  • ❌ 浏览器 tab——需要切换 tab 才能看，离手成本高且容易被关掉
  • ❌ 只做副屏不做快捷键——使用者 80% 时间没有/不开副屏，工具就废了
  • ❌ 只做快捷键不做副屏——失去"持续在场的状态指示器"价值

哲学四：使用即沉淀

每一次点击、每一次复制、每一次组合，都应该成为资产的一部分，而不是挥发掉的操作。

传统工具把"沉淀"当作一个独立动作（手动保存、手动命名、手动分类）——这个摩擦让 90% 的有价值信息都没被沉淀下来。本项目的设计目标：让沉淀成为使用的副产品。

设计上的含义：每次调用自动记录 usage_count、last_used_at；重复使用的组合自动识别并提示保存；使用热度数据驱动月度 review，而不是靠回忆。

哲学五：话术是分层语言系统

经过观察，一个成熟的提示词库不是扁平的列表，而是分层的语言系统：

Modifier（原子）
    ↓ 组合
Composition（组合）
    ↓ 固化
Macro（宏定义 / Favorites）
  • Modifier（原子）：不可再分的方法论动作，比如"第一性原理思考""启动子代理并行"
  • Composition（组合）：由多个 Modifier 按顺序拼接而成的完整提示词
  • Macro（宏）：高频使用的 Composition 被固化为单独的快捷入口

设计上的含义：三层结构必须在 UI 上都有对应位置；三层之间的跃迁路径必须被设计（Composition → Macro 的固化、Modifier → Composition 的组装）；数据模型也必须支持这种跃迁的可追溯性（一个 Macro 由哪些 Modifier 组成？）。

哲学六：减少"离手"但不消除

"离手"指的是：注意力从主屏（和 AI 对话）离开，去操作辅助工具。

  • 完全消除离手 = Raycast 式浮层 → 属于自动挡阶段
  • 完全不管离手 = 频繁切屏、手忙脚乱 → 过度原始

手动挡阶段的正确选择：把离手动作显式化、标准化、最小化。这种受控的离手，反而创造了思考的刹车片——点击前那一瞬间的再确认，让误用率下降。

哲学七：人机协同需要协议对齐

和 AI 高效协作的前提，不是"我有多好的任务提示词"，而是AI 知道我这次协作的模式是什么、要产出什么形态的东西。一次清晰的相位对齐，胜过十句精雕细琢的任务话术。

对齐话术定义"框"，任务话术在框里工作。框不对，任务话术再精都无效；框一旦定下来，后续沟通摩擦大幅下降，输出形态可预测。

因此对齐话术是协作的元层——它不参与任务话术的组合、不被 SOP 引用、不和 Macro 互相转化。它独立成对象（AlignmentPhrase），独立成 UI 区域（相位带），独立成认知动作。

哲学八：双形态承载，按场景分工

仪表盘不存在"唯一正确的承载形态"——使用者一天的工作里，有些瞬间需要召之即来的工具，有些瞬间需要持续在场的状态指示。强行用一种形态覆盖两种需求，结果是两边都不顺手。

维度主形态：快捷键唤起辅形态：副屏常驻
使用占比80%（日常高频）20%（深度场景）
触发方式全局快捷键永远在场
退出方式复制后自动隐藏 / ESC不退出
核心价值高效查找、快速调用全景巡视、状态监视
典型场景日常对话中复制话术月度 review、整理资产、长任务跟随
优化方向召之即来用完即走持续可见余光感知
设备依赖任何桌面（含笔记本单屏）需要副屏硬件

两种形态共享同一套数据后端、同一套 UI 资产——它们不是两个产品，是同一个产品的两种唤起方式。使用者切换形态的成本应该为零。实施时应先做主形态 MVP，再做辅形态。

哲学九：界面自身也是被维护的资产

真实仪表盘对应物：赛车手会根据自己的习惯调整仪表盘布局——哪些信息放哪里，是可以被个人化配置的。

决策：界面本身必须可配置——Scene 的排序、显隐可调；Phase 的增删、排序、改色可调；Macro 的置顶数量可调；SOP 模板可以自由增删；Modifier 池可编辑、增删、分组重命名；首屏六个区域的高度比例可调。

反设计：把配置项藏在设置页（常用的配置应该在首屏就能调）；用复杂的设置系统（每个配置项都应该是"点一下就切换"的简单开关）。

2.3 核心概念

三层资产模型

这是整个项目最核心的数据模型。其他所有设计都围绕这个模型展开。

Modifier（方法论原子）

定义：不可再分的方法论动作，是其他所有话术的"构造材料"。

  • 通常以"动宾短语"形式存在，比如"主动调研、推进，""从更高的视角重新设计，"
  • 末尾常带逗号而非句号——它是"半句话"，设计上就是要被拼接的
  • 可按作用类型分组：认知前置（思维）、动作策略（执行）、产出形态（交付）、通用约束（约束）

设计要求：每个 Modifier 必须能独立成立，不依赖上下文；Modifier 之间应尽量正交，避免语义重复；数量控制在 30 以内——过多会稀释组合的价值。

Composition（组合）

定义：由多个 Modifier 按固定顺序拼接而成的完整提示词，服务于某个具体任务场景。

标准结构（推荐顺序）：

[角色设定] + [认知前置 Modifier] + [核心任务] + [动作策略 Modifier]
+ [产出形态 Modifier] + [通用约束 Modifier]

示例：作为战略合伙人，（使用第一性原理思考，）全面负责本次方案的完善，（主动调研、推进，搜索最佳实践或 GitHub 开源实践作为参考，）减少实施路径，（输出经得起推敲的方案，先出方案供我拍板。）

Macro（宏定义 / Favorites）

定义：高频使用的 Composition 被固化为单独的快捷入口，绕过组合过程直接调用。

  • Composition 是现场组装的（临时拼接）
  • Macro 是预先固化的（一键调用）

Macro 的来源：90% 应来自"曾经的 Composition"——使用者发现某个组合反复出现，就把它升级成 Macro；10% 可以是"原生的 Macro"——不由 Modifier 组合而来，而是直接写出来的成品话术。

典型示例："推翻重来" / "借力最优解" / "经得起推敲的方案" / "第一责任人主动找事做"。

场景（Scene）

定义：按"使用时机"对话术进行的横切分类。示例：规划（plan）、调研（research）、方案（debug）、排查、部署、审查、日常、工作流……

Scene 是正交维度。一条 Modifier 可以属于某个 Scene，也可以跨 Scene 通用。Scene 数量控制在 10 个以内；Scene 之间必须互斥；Scene 可以有子状态（如"方案"Scene 包含"生成→评审→修订→定版"四个子阶段）。

SOP（标准作业流程）

定义：由多个 Macro 或 Composition 按时间顺序串联起来的工作流模板。

  • Composition 是单次对话内的提示词组合（横向拼接）
  • SOP 是跨多次对话的话术使用序列（纵向串联）

示例 SOP（方案设计型）：

1. /借力最优解              → 让 Claude 先调研外部最佳实践
2. /全局视角出方案          → 基于调研结果出整体方案
3. /多 agent 评价方案       → 多 Agent 交叉评审
4. /根据 codex 意见修改     → 收敛修订建议
5. /更新方案                → 整合修订成最终版

SOP 是这个项目第一个"超越单次对话"的概念——它体现了使用者的工作流节奏，也是未来迁移到自动挡阶段的"半成品蓝图"。SOP 不引用 Modifier（颗粒度不够）、不引用 AlignmentPhrase（哲学七边界）。

三个关键数据指标

  • usage_count：每条话术 / Modifier / Macro 被复制的次数，永久累加
  • last_used_at：最近一次被复制的时间戳
  • session_recent：本次会话中最近 5 次复制记录（用于"刚刚操作了什么"）

这些指标不是锦上添花，是驱动哲学四（使用即沉淀）落地的核心机制。没有它们，这个项目就退化回普通的"提示词陈列馆"。

认知相位（Phase）与对齐话术（AlignmentPhrase）

使用者在长期 AI 编程实践中观察到：一次完整的人机协作任务，无论 Scene 是什么，内部都会经历一组认知相位——发散、理解、规划、生成、执行、收敛、沉淀、迭代。每个相位对 AI 的期待完全不同。

当前 Phase 实践版本（可配置增删）
相位协同期待
发散铺开可能性，不下结论
理解拆解问题、找出本质
规划制定路径和优先级
生成产出具体方案/代码/内容
执行按既定方案推进，少质疑多动手
收敛对比、评估、给出结论
沉淀复盘、提炼、记录
迭代基于反馈修订，推翻重来
AlignmentPhrase（对齐话术）

定义：人机协作的协议层话术。在任务开始前、或对话中途切换相位时使用，告诉 AI 当前的认知相位，定义本次协作的"框"。

示例：

  • 发散相位："我们现在做发散，先把所有可能性铺出来，不要给结论，不要评估优劣。"
  • 收敛相位："我们现在做收敛，从前面的方案里选出最优解，给出明确推荐和理由。"
  • 沉淀相位："我们现在做复盘，目标是提炼可复用的方法论，不是重新评估方案本身。"

独立成对象——不寄生在 Macro 里；属于 Phase——每条 AlignmentPhrase 必须归属于一个 Phase；孤岛声明——不参与 Modifier 拼接、不属于任何 Scene、不被 SOP 引用、不和 Macro 互相转化。

2.4 不做什么（Non-Goals）

这一章比"做什么"更重要。明确拒绝的事情，决定了这个产品的性格。

8.1 不做 AI 生成

  • 不内嵌任何 "用 AI 帮我生成 Modifier" 的功能
  • 不内嵌 "AI 自动分类我的话术" 的功能
  • 不做 "AI 自动建议 SOP" 的功能
  • 不做 "AI 自动判断当前相位" 的功能

理由：引入 AI 生成会让使用者失去"自己思考"这一步，而这一步恰恰是手动挡阶段的核心价值。本工具的目的是沉淀使用者自己的方法论，不是让使用者消费 AI 生成的通用话术。

8.2 不做社区 / 分享功能

不做"导出为可分享的链接" / "浏览他人的提示词库" / "团队协作"。理由：单人工具的价值最大化依赖高度个性化；使用者的话术沉淀涉及其工作方式的指纹——分享机制反而产生隐私压力。

8.3 不做自动执行

  • 点击话术只复制到剪贴板，不自动发送给任何 AI
  • 不集成 Claude API 直接对话
  • 不做"一键跑完整个 SOP"的自动化
  • 不做"切换相位时自动发对齐话术给 Claude"——必须人主动粘贴

理由：手动复制粘贴虽然多一步，但这一步是思考的缓冲，有认知价值。

8.4 不做多层级文件夹

Scene 下不允许再嵌套子 Scene；Macro 不允许分组到文件夹；Modifier 只有四个固定分组；Phase 下不允许再嵌套子 Phase。理由：层级 > 1 就会变成"找东西"——违背哲学二。

8.5 不做付费 / 商业化

这是自用工具，不考虑商业化路径。理由：目标是沉淀使用者自己的方法论，商业化会扭曲设计方向。

8.6 不让对齐层与任务层混淆

协议层（AlignmentPhrase / Phase）和任务层（Modifier / Composition / Macro / Phrase / Scene / SOP）必须物理分离：

  • AlignmentPhrase 不出现在 Composition 工作台（不参与 Modifier 拼接）
  • AlignmentPhrase 不能被加入 SOP 步骤
  • AlignmentPhrase 不与 Macro 互相转化（即使内容相似也是两个对象）
  • Macro 区不展示 AlignmentPhrase
  • 相位带（Phase Bar）不展示任何任务话术
  • Scene 不包含 AlignmentPhrase

8.7 不让主辅形态各自走偏

不做"主形态独占功能"和"辅形态独占功能"的对立；不做两套独立 UI；不做数据同步——两种形态读写同一份本地数据，零同步层；不做主形态和辅形态之间的数据迁移——它们就是同一份数据。

2.5 开放问题（来自 spec §10）

这些是 spec 当前无法确定、需要在实施过程中根据真实使用反馈决策的问题。

  • 10.1 Modifier 池的增长管理——什么时候该新增 / 修改 / 拆分 / 合并？观察期：使用 3 个月后看 usage_count 分布。
  • 10.2 SOP 的粒度——SOP 步骤应该是"一次复制操作"还是"一个对话阶段"？观察期：第三阶段实施后看实际使用情况。
  • 10.3 手动挡 → 自动挡的过渡点——什么时候应该把某个 SOP 升级为 Claude Code 的 composite command？观察期：有至少 2 个 SOP 被使用超过 20 次之后。
  • 10.4 Phase 体系的稳定性——当前 8 个 Phase 是否稳定？观察期：MVP 使用 1-2 个月后看 phase_id 分布。
  • 10.6 Scene 区在 user flow 中的存在感——三个典型场景 Scene 区都没作为主角出场；可能更适合"事后回看"角色。当前决策：暂不调整。
  • 10.7 搜索语法演化——是否需要 #macro #scene:方案 等过滤前缀？当前决策：先不做语法。
  • 10.8 双形态的使用比例验证——主辅 80/20 假设未经长期验证。观察期：辅形态实现后至少使用 1 个月。

8 条不可违反铁律 · 违反任一条即转型（🧑 人主笔）

3.1 八条铁律

每条都「违反即死」——违反任一条，项目就不再是 prompt-hub。修订必须开 ADR。本文件 6 个月内不应频繁修订。

A1 · 桌面原生 only

本项目必须是桌面原生应用（Tauri / Electron 类），不做 Web 版、浏览器扩展、移动端。

  • 证伪：见到 web build target 或 PWA manifest → 违反
  • 依据：哲学三时间分离机制依赖原生快捷键 + 全屏覆盖窗口，浏览器不可达
  • 后果：失去主形态承载机制，退化为 web 提示词管理器（市场已饱和）

A2 · 本地优先 + 无服务端

所有数据存本地（localStorage / SQLite / 本地文件），不调用自有后端，不依赖云同步。

  • 证伪：见到 fetch 到自有 API / 注册服务端依赖 → 违反
  • 依据：单人工具 + 话术含工作指纹 + spec §8.2 隐私论证
  • 后果：引入运维 / 网络依赖 / 隐私风险——三者任一让产品转型

A3 · 单人单机

不引入用户系统、权限模型、协作机制、分享链接。

  • 证伪：见到 user_id / OAuth / share token → 违反
  • 依据：单人工具的价值最大化依赖高度个性化
  • 后果：设计向"通用好用"妥协，失去个性化深度

B1 · 资产三层模型不可破

仅 Modifier / Composition / Macro 三层，不引入第 4 层资产类型。

  • 证伪：见到 SuperMacro / SubModifier / 任何第 4 类资产 schema → 违反
  • 依据：三层是观察沉淀的语言系统；第 4 层破坏组合可追溯性
  • 后果：数据模型膨胀，UI 无展示位，组合语义混乱

B2 · 协议层与任务层物理分离

AlignmentPhrase 不参与 Modifier 拼接、不入 SOP、不与 Macro 互转、不归属 Scene。

  • 证伪：见到 SOP 步骤含 AlignmentPhrase / Macro 区展示 AlignmentPhrase → 违反
  • 依据：哲学七「先定框、再发任务」的认知节奏依赖物理分离
  • 后果：失去元层协议握手能力，退化为更花哨的提示词管理器

C1 · 主形态快捷键唤起 ≤ 200ms 可见

全局快捷键按下到窗口完整可交互必须 ≤ 200ms（P95）。

  • 证伪：benchmark P95 > 200ms → 违反
  • 依据：哲学三「召之即来」——超 200ms 产生迟滞感，主形态心智崩溃
  • 后果：使用者放弃主形态，工具退化为偶尔打开的辅形态

D1 · 不内嵌 AI 生成

工具内部不调用 LLM 生成 / 优化 / 分类 / 推荐话术（用户手动粘贴到外部 AI 不算）。

  • 证伪：见到 anthropic / openai / 任何 LLM SDK 用于生成话术 → 违反
  • 依据：手动挡阶段的核心价值是使用者自己思考
  • 后果：使用者消费 AI 通用话术，失去沉淀自己方法论的机会

E1 · 宪法变更必须开 ADR

本文件任何一条铁律的修改 / 删除 / 新增，必须先开 ADR 决议；PR 描述需链接 ADR 编号。

  • 证伪：constitution PR 无 ADR 链接 → 违反
  • 依据：宪法变更影响整个项目方向，必须留下决策痕迹
  • 后果：宪法沦为可随意改写的「建议」，失去约束力

UI 契约 · 双形态架构 · 一屏全景 · 三个核心 user flow · 主形态 UI 草案（🤝 共创）

4.1 双形态架构

两种形态的关系

本项目采用单一应用、双形态承载架构：

  • 主形态：快捷键唤起的全屏覆盖窗口——80% 使用场景
  • 辅形态：副屏常驻视图——20% 深度场景

关键性质：一个桌面应用，两种形态由同一个 Tauri 应用提供；共享数据后端，零同步成本；共享 UI 模块，只是容器布局不同；形态切换无成本。

主形态的核心特征

  • 触发：全局快捷键（默认 ⌥ Space，可配置）
  • 承载：全屏覆盖窗口，半透明背景（透明度约 92%），主屏内容可透出感知
  • 退出：复制话术后自动隐藏；ESC 手动隐藏；点击窗口外区域隐藏
  • 使用画面：在主屏想事情 → 按下快捷键 → 仪表盘全屏覆盖 → 扫视/搜索定位话术 → 点击或回车复制 → 窗口自动隐藏 → 回到主屏粘贴

辅形态的核心特征

  • 触发：应用启动时常驻于副屏
  • 承载：副屏常驻窗口，不覆盖主屏
  • 退出：不退出（关闭应用即关闭副屏视图）
  • 使用画面：月度 review / 整理资产 / 跟随长 SOP 时 → 抬头看副屏 → 在副屏点击/编辑 → 主屏继续工作

UI 共用规则

两种形态使用同一套 UI 模块，区别只在容器布局：

模块主形态辅形态
搜索框顶部居中，中等大小顶部居中，中等大小
相位带搜索框下方，8 个横排搜索框下方，8 个横排
Macro 区中部左侧，60% 宽中部左侧，60% 宽
Scene 区中部右侧,40% 宽中部右侧，40% 宽
最近使用底部左侧底部左侧
SOP 导航底部右侧底部右侧
状态栏最底部最底部

差异点：主形态背景半透明、有"复制即隐藏"动画、ESC 关闭；辅形态背景不透明、不会隐藏、可选展开 Composition 工作台为侧栏。

3.2 一屏全景布局原则

真实仪表盘对应物：汽车仪表盘上，时速表、转速表、油量、水温、挡位指示灯同时可见，驾驶员不需要切换视图。

决策：仪表盘首屏（两种形态都适用）必须同时呈现以下七个区域：

  1. 搜索框（顶部居中，中等大小）——全局兜底入口，⌘K 唤起聚焦
  2. 相位带——8 个 Phase 横排，承载 AlignmentPhrase 一键调用
  3. Macro 区——当前所有 Macro 作为卡片墙置顶
  4. Scene 区——按场景横切的话术分类（Tab 切换）
  5. 最近使用区——最近 5 次复制历史（时间倒序）
  6. SOP 导航区——当前激活的 SOP 模板 + 进度指示
  7. 状态栏——今日复制次数、当前相位、未分类草稿等指标

为什么搜索框是"兜底"而非"主路径"：哲学二要求扫视为主、搜索为辅；搜索是"知道有这条话术但记不清在哪"时的兜底；搜索框视觉权重刻意压低（背景浅灰、占位文字、约 60% 宽度居中），不诱导优先使用。

反设计：搜索框作为主导航（Raycast、Alfred）——隐藏了全局信息；瀑布流式——需要滚动才能看全；主导航 Tab 式——把同类信息藏在不同 Tab 里。

3.3 点击路径按频率分层

真实仪表盘对应物：换挡杆的挡位布局——1、3、5 挡在上排（常用），2、4、R 挡在下排（次常用）。使用者按频率找挡位，不是按字母顺序。

Macro首屏直达 → 点击即复制1 步
最近使用首屏直达 → 点击即复制1 步
Scene 内话术点 Scene → 点话术2 步
Composition 组合进入组合模式 → 勾选 → 调序 → 复制3-4 步

3.5 状态反馈：刚刚发生了什么

任何操作都必须有即时的视觉确认：

  • 复制成功：话术卡片闪一下绿色 + 角落显示"已复制"
  • 相位切换：相位带高亮当前相位 + 状态栏显示"当前相位：X"
  • 组合添加：组合队列区出现新条目 + 该 Modifier 卡片变色
  • Macro 保存：Macro 区顶部出现新卡片 + 轻微上浮动画
  • 使用次数更新：话术卡片右上角的数字 +1
  • 拆解视图展开：Macro 长按 / 右键展开"由哪些 Modifier 组成"

3.6 用户旅程（User Flow）

传统 App 的 user flow 是"用户从 A 屏点到 B 屏完成 X 任务"。本项目的 user flow 不只是 UI 跳转——它必须把"人机协同的认知节奏"画出来。一次完整任务横跨主屏（Claude Code 对话）+ 仪表盘 + 头脑（认知动作）三个空间。

下面 3 个核心场景按主形态视角描写。辅形态差异：不需要"按下快捷键"——副屏永远在场；复制后窗口不会自动隐藏——副屏持续呈现；余光感知关键状态。

场景一：完整 SOP 驱动型（方案设计任务）

任务背景：接到新需求"给现有系统加一个数据导出模块"。已有"方案设计型 SOP"模板可以驱动整个流程。

叙事

我在主屏的 Claude Code 里打开了项目，光标停在对话框里。但我没有立刻打字——我先抬头看副屏。

副屏的相位带最顶部，8 个 Phase 横排着。我此刻要做的是方案设计，第一步应该是调研外部最佳实践——这是"发散"相位。我点了一下"发散"，剪贴板里立刻有了那条对齐话术。我回到主屏，Cmd+V 粘贴，回车。

Claude 收到了对齐话术，回了一句"好，我们现在进入发散模式，先不下结论"。

然后我再次抬头看副屏——SOP 导航区已经识别出我刚才用的话术属于"方案设计型 SOP"的第 1 步。我点击 SOP 中的第 1 步，剪贴板里换成了"借力最优解"这条 Macro。回到主屏粘贴，加上我的具体任务描述。

Claude 给我铺开了 5 个方案。看完，发现已经够发散了，该收敛了。抬头看副屏——状态栏显示"当前相位：发散"。我点击相位带的"收敛"，剪贴板里有了收敛对齐话术，主屏粘贴。SOP 导航区同步推进到了第 2 步。

任务结束前，副屏弹出一个不打扰的提示："你今天用了一个新组合 3 次，是否保存为 Macro？" 我点确认，那个组合成了 Macro 区的新卡片。

5.1 跨形态切换（主 ↔ 辅）

场景：用户在副屏开着辅形态做月度 review，主屏正在和 AI 对话，忽然需要快速复制一个 Macro。

用户在主屏对话 → ⌥ Space → 主形态弹出（覆盖主屏）
  → 点 Macro 复制 → 主形态自动隐藏（哲学六）
  → 辅形态副屏状态同步更新（usage_count +1）
  → 用户继续 review，看到刚才用的 Macro 高亮在「最近使用区」

关键约束：主辅形态共享同一数据后端；主形态复制触发 UsageRecord 写入 → 辅形态 reactive 监听更新；辅形态不要因主形态唤起而隐藏。

5.2 升级迁移（schema_version 演进）

场景：用户使用 v1.x，应用升级到 v2.0（major 升级）。

启动 → 检测 schema_version=1.1 vs 代码内置 2.0
  → 显式弹窗：「v2.0 即将到来 / 影响范围 / 会丢失字段 / 是否继续」
  → [取消] → 保持 v1.x 二进制继续运行
  → [继续] → 强制要求导出 v1.x 备份到指定位置
    → 导出成功 → 执行 migrate_1_x_to_2_0
      → 迁移成功 → schema_version=2.0，记录 migration_log
      → 迁移失败 → 回滚到备份 → 错误提示 + 用户介入
    → 导出失败 → 中止迁移，保持 v1.x

约束：minor 升级零感知；major 升级强同意；任何迁移前自动备份。

5.3 数据导入失败

用户从其他工具导入 JSON 数据，格式不兼容或含恶意字段。

view:data-io → 选择 JSON → 导入
  → schema 校验
    → 通过 → 写入本地数据库 → 提示「成功导入 N 条」
    → 失败 → 显示具体错误 → 提供「查看示例 schema」
            → 数据库保持导入前状态（事务回滚）

关键约束：禁拒键 __proto__ / constructor / prototype（防原型污染）；单条话术 ≤ 5000 字符；总文件 ≤ 10 MB；必须显示具体错误，不允许"导入失败请重试"模糊提示。

5.4 删除资产的 ask-first flow

用户在配置面板尝试删除一个使用过 48 次的 Macro，弹三选项：取消 / 标记为 deprecated（保留历史）/ 永久删除。三选项必须始终给（不允许"取消/确认"二选一）；高频使用资产（usage_count ≥ 10）默认推荐 deprecated 而非删除。

引导页 ≤ 3 屏（定位 + 三层资产 + 是否导入示例）；不强制导入示例；不弹出教程视频；不要求账号注册。

6.1 设计原则

真实仪表盘对应物：汽车仪表盘的字体和图标都比普通显示器大，颜色对比度更高——因为驾驶员是扫视，不是阅读。

核心决策：UI 参数比常规网页放大一档，针对"扫视为主、阅读为辅"场景优化。使用者需要在 1 秒内识别屏幕上的关键资产。

6.4 对比度（WCAG 实测）

总原则：主文字 vs 背景 ≥ 7:1（WCAG AAA）；非文字元素（边框 / 图标 / 控件）≥ 3:1；禁止依赖颜色单一维度传递信息——必须颜色 + 形状 + 位置多维度冗余编码。

5.0 搜索区

5.2 Macro 快捷区

定位：首屏第二显眼位置（仅次于相位带），承载使用者最高频调用的任务话术。

内容：使用者手动标记为 Macro 的话术；每张卡片显示标题 / 内容预览（2-3 行）/ 使用次数 / 最近使用时间；卡片按使用频次排序（可切换为手动排序）。

关键交互：单击 → 复制 + 视觉反馈；长按 / 右键 → 打开"拆解视图"（展示 expandFrom Modifier）；拖拽 → 调整顺序；卡片右上角 "…" → 编辑 / 删除 / 移入 Scene / 标记 deprecated。

5.3 Scene 全景区

定位：按"使用时机"横切的话术分类展示。

内容：10 个以内的 Scene 卡片，显示 Scene 名称和图标 / 话术数量 / 前 3-5 条话术标题预览 / 本周使用次数。点击 Scene 后在首屏展开该 Scene 的全部话术（不跳页）。如果 Scene 有 sub_stages[]（如"方案"Scene 的四阶段），按子阶段分组显示。

5.4 Composition 组合工作台

定位：把 Modifier 按顺序拼装成完整 Composition 的工作区域。

内容：Modifier 池（20-30 个，按四组排列）；组合队列（被选中的 Modifier 按顺序排列）；角色和任务输入框；实时预览。

关键交互：单击 Modifier → 加入组合队列；队列中的 Modifier 可拖拽排序；填写角色和任务后预览实时刷新；按钮：复制 / 保存为 Macro / 保存为 Scene 话术 / 清空。

边界：AlignmentPhrase 不进入组合工作台——协议层与内容层物理分离。

5.5 最近使用区

定位：短期记忆窗口，承载本次工作会话的复制历史。

内容：最近 5 条复制记录（时间倒序），每条显示内容预览（一行）、来源（Macro/Scene/Composition/Alignment）、时间。

关键交互：单击 → 再次复制（最常见用途：刚才粘贴错了）；长按 / 右键 → 把该条话术升级为 Macro（AlignmentPhrase 类型不提供"升级为 Macro"选项）。

实现注意：本地 session 范围，不跨会话保留；5 条是上限，再多就成了"历史记录"而非"最近使用"。

5.6 SOP 导航区

定位：跨对话的工作流指引，是整个仪表盘中"向前看"的部分。

内容：当前激活的 SOP 模板（带序号的步骤列表 / 流程图）；进度标记（已完成的步骤打勾，当前步骤高亮，下一步突出）；SOP 模板切换器；"新建 SOP"按钮（从使用历史录制）。

SOP 模板的初始设计：方案设计型 / 调研型 / 排查型 / 复盘型。SOP 模板应该是可自定义的。

重要边界：SOP 步骤不引用 AlignmentPhrase——相位切换由使用者在执行 SOP 过程中手动通过相位带完成，不混进 SOP 步骤序列。

5.7 状态仪表区

定位：量化使用数据，驱动迭代。

内容：今日复制次数 / 今日最常用 Top 3 / 本周新增 Macro 数 / 当前所处相位 / 今日各相位停留次数分布 / 未分类草稿数 / 提示词总数的变化趋势。

这是哲学四（使用即沉淀）的收口模块——让沉淀从"默默发生"变成"显式可见"。"未分类草稿"是整个系统里最宝贵的信号——它告诉使用者哪些高频使用尚未沉淀。

实现注意：数字要大、显眼，不是装饰性小字；不要用花哨的图表；首次启动空状态需引导文案，不能白屏。

5.8 配置与迭代入口

入口位置：右上角小齿轮图标。

可配置项：Scene 增删改 / Phase 增删改 / Modifier 池增删 / AlignmentPhrase 批量编辑 / SOP 模板增删 / 首屏区域比例 / 数据导入导出（JSON）。

4.3 非功能需求（PRD §7）

7.1 性能

  • 首屏加载时间：≤ 1.5 秒（冷启动）
  • 任何点击响应：≤ 100 毫秒（含复制、搜索、切 Scene、切 Phase）
  • 支持数据量：Modifier 30 条、Macro 100 条、Scene 10 个、Phase 12 个、AlignmentPhrase 50 条、SOP 20 个、Phrase 总计 300 条——超过这个规模说明资产膨胀需要清理

7.2 离线可用 · 7.3 隐私

  • 完全本地运行，不依赖任何后端服务
  • 数据持久化在 SQLite / 本地数据库，不上传到云端
  • 不内置任何第三方分析脚本（GA / Sentry 等一律禁用）
  • 不调用任何外部 API（不做话术的 AI 辅助生成）

理由：使用者涉及敏感业务场景（政府/公安数据方向），任何云端存储都是硬伤。本地优先是硬性要求。

7.4 跨设备

  • 主要设备：Mac/Windows 桌面（主形态：快捷键唤起 + 全屏覆盖；辅形态：副屏常驻）
  • 次要设备：iPad（横屏，无副屏条件或便携场景；仅作辅形态备选）
  • 辅助设备：iPhone（移动场景下临时查看，只读为主）
  • 不做多设备实时协同——单人使用，最终一致性即可

7.5 可迁移性

最重要的非功能需求之一：使用者应该随时可以把数据拿走、换个工具继续用。所有数据以 JSON 格式存储；提供"导出全部数据"按钮；提供"从 JSON 导入"功能；数据格式保持稳定（向后兼容）。

7.7 升级回滚契约

schema_version 采用 major.minor 字符串格式。

升级类型触发用户感知
minor 升级向后兼容的字段增加 / 索引调整 / 可空字段引入无感（旧数据可直接读）
major 升级字段删除 / 重命名 / 类型变更 / 必填字段引入显式弹窗 + 强制迁移 + 旧版导出兜底

升级流程 6 阶段：检测 → 备份（自动 JSON 导出）→ 迁移（minor 自动 / major 用户确认）→ 验证 → 完成 → 记录迁移日志。

设计原则：minor 升级零感知；major 升级强同意；任何升级都可回滚（自动备份是硬约束）；schema_version 是真理之源——任何代码逻辑都通过读 schema_version 决定如何处理数据，禁止"猜版本"。

4.5 安全 / 合规字段（PRD §9）

按「完全本地+离线」档触发——必填：A 输入验证 + C 敏感数据销毁；不必：B 认证 / D 速率限制 / E 日志审计 / F 加密 / G 供应链。

A 输入验证（MVP 必填）

  • InputSchema：所有用户输入必须长度限制 + 字符集限制
  • MaxSize：单条话术 ≤ 5000 字符；总数据导出 JSON ≤ 10 MB
  • AllowedTypes：文本（UTF-8）+ JSON（数据导入）+ 标准 markdown 符号
  • SanitizationRule：XSS 过滤（render 时 escape）；JSON 导入时 schema 校验（拒绝 __proto__ / constructor 等键）

C 敏感数据销毁（MVP 必填）

  • DeleteTrigger：用户主动删除 Modifier/Macro/Phrase + 配置面板"清空所有数据"
  • ScrubMethod：localStorage / IndexedDB 物理删除 + 内存中变量置 null + 清空剪贴板（敏感复制后 30 秒）
  • VerifyDeletion：删除后检查 localStorage 不含相关 key + 内存释放

5.2 第一阶段：主形态 MVP

目标：跑通"快捷键唤起 → 对齐 → 看全景 → 一键复制 → 自动隐藏"这个核心闭环。

技术形态

  • 桌面运行时：Tauri 2.x + macos-private-api feature
  • 前端框架：React 19.2 + TypeScript 5.x strict
  • 状态管理：Zustand 5 — appStore 管窗口 visibility / promptStore 管 Macro/UsageRecord / searchStore 管 query / settingsStore 管快捷键与 Phase
  • 数据层：rusqlite 0.32 + bundled SQLite；数据目录 ~/Library/Application Support/dev.prompt-hub/
  • 样式：CSS Modules + CSS variables（token 全部引用 §0 T1 根样式表）
  • 测试：Vitest 4 + Testing Library + jsdom 29 跑前端单元；cargo test + tempfile 跑 Rust + SQLite fixture
  • 全局快捷键注册（默认 ⌥ Space，可配置）
  • 主形态：全屏覆盖窗口 + 半透明背景（约 92%）+ NSWindow level 浮于所有应用上方但不抢焦点（macos-private-api）
  • 复制即隐藏 + ESC 关闭

包含的功能模块

  • 搜索区——⌘K 聚焦，全局兜底
  • 相位带——⌘1-⌘8 切相位
  • Macro 快捷区——按热度排序的卡片墙
  • Scene 全景区——Tab 切换
  • 最近使用区——5 条历史
  • 复制和使用计数（UsageRecord）

不包含：Composition 工作台、SOP 导航、状态仪表、配置入口、副屏形态。

为什么先做这些

  • 80% 使用场景由"主形态 + Macro + 相位带"覆盖——这部分跑通就比现有工具更好用
  • 哲学七（协议对齐）是本项目最特别的设计，必须在 MVP 阶段就能验证
  • 辅形态（副屏）依赖主形态的所有模块——主形态先稳，辅形态自然顺

交付标准

  • 按下快捷键 0.3 秒内仪表盘弹出
  • 点击/回车复制成功，窗口在 200ms 内隐藏
  • 任何复制动作都被记录到 UsageRecord
  • 相位切换有明显视觉反馈
  • ⌘K 搜索能在所有 Macro 和 Phrase 中找到目标

5.3 第二阶段：闭环沉淀

目标：实现"使用即沉淀"的哲学。

包含：Composition 工作台（通过 ⌘N 唤起子窗口）；状态仪表区——含相位分布数据；"未分类草稿"识别和提示；"保存为 Macro"的自动提示。

交付标准：使用者重复使用某组合 3 次后，下次唤起时窗口顶部弹出"是否保存为 Macro"提示。状态仪表能展示相位停留分布。

5.4 第三阶段：SOP 导航

目标：引入"时间"维度，为未来的自动挡做铺垫。

包含：SOP 导航区——首屏底部右侧；SOP 模板的创建和编辑；从使用历史录制 SOP。

交付标准：能按照预设 SOP 一步步执行，每步完成后高亮下一步。

5.5 第四阶段：配置与个性化

目标：界面本身成为被维护的资产。

包含：配置入口——含 Phase 可配置编辑；数据导入导出；主形态界面布局的可配置性。

交付标准：使用者可以在不碰代码的前提下，完整地调整这个工具以适配自己的工作方式变化，包括增删 Phase、调整快捷键、修改 Scene 结构。

5.6 第五阶段：辅形态（副屏常驻视图）

目标：扩展辅形态，覆盖 20% 的深度场景。

包含：副屏常驻窗口（应用启动时打开，独立窗口模式）；共享主形态的所有 UI 模块；月度 review 视图（基于 UsageRecord 的长期趋势图）；副屏专属：状态持续可见（不会隐藏）；副屏专属：Composition 工作台可作为侧栏常驻。

交付标准：副屏窗口能呈现完整全景，与主形态 UI 一致；数据完全共享，主形态做的任何修改在副屏即时可见；关闭副屏窗口不影响主形态使用。

为什么放最后

  • 辅形态是主形态的"翻版"，UI 代码可以大量复用
  • 主形态需要充分验证后，才能确定辅形态需要哪些"专属优化"
  • 副屏不是所有使用者都有——先把主形态做扎实，覆盖最广人群

ADR-001 后：Tauri 2.x 实测内存常驻 50-100MB / 包体 10-20MB，远低于上述上限。本表保留为警戒线（regression 防线），非目标——若实际数字接近上限说明出现内存泄漏或资源未压缩。
