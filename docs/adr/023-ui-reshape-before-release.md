# ADR-023: v0.1.0 发布前进行 UI + 组件架构系统性重塑

## 1. 标题与日期

- **标题**：将排版收敛（原批次 C）、响应式底线（原批次 D）、动效语言、组件架构拆分合并为一次连贯重塑，在 v0.1.0 发布前完成
- **日期**：2026-07-21
- **决策者**：omar（口头裁决，AI 记录）
- **影响范围**：src/ 前端全量、design-spec §9、2026-07-12-ux-taskflow-audit §6 批次排序、HANDOFF 发布节奏

## 2. Status

`Accepted`（2026-07-21，omar 会话内两问拍板：范围 = UI+组件架构；时机 = 重塑完成后再发）

## 3. Context

- **触发事件**：omar 真机使用后反馈"使用体验、视觉感受不是很好"，且认为"改动太零散"——问题池销账式打补丁缺乏整体设计意图
- **live 证据**：2026-07-21 截图证实暗 band（ADR-020）在 light 模式下未渲染为深色岛，协议/任务层视觉分离失效；elevation 几乎不可感知；排版无扫视锚点
- **业务约束**：宪法 C1（200ms 唤起）、B1/B2（三层模型 + 协议/任务分离）不动；Rust 层与数据契约不动
- **不决策的代价**：继续按 D-6 排序发布 v0.1.0，第一印象由未收敛的视觉承担

## 4. Options Considered

### Option A: 维持 D-6 —— 先发 v0.1.0，批次 C/D 发布后做
- **优点**：发布节奏不延误；已收口批次 A/B 尽快见用户
- **缺点**：首发观感差；补丁式节奏延续，"零散感"无解

### Option B: 发布前一次连贯重塑（UI 视觉体系 + 组件架构），完成后再发 ✅
- **优点**：设计体系（ADR-018/019/020 已定方向）被忠实落地而非继续打折；架构拆分与 CSS 收敛同窗完成互不返工；首发即"优秀"水位
- **缺点**：v0.1.0 发布时间后移；单窗口改动面大，依赖 464 条测试兜底

### Option C: 全量重写（含 Rust 层）
- **缺点**：丢弃无痛点且全绿的后端与行为契约，风险纯增无收益

## 5. Decision

> **一句话拍板**：选 Option B——发布前完成 UI 层系统性重塑 + 组件架构拆分，Rust 层与数据契约不动；本 ADR supersede 审计 D-6 中"批次 C/D 放发布后"的排序（D-6 其余内容仍有效）。

**为什么不选其他**：
- 不选 A：零散感的根因是缺整体设计 pass，发布后再修等于把最差的第一印象送出去
- 不选 C：重写的恰是没有痛点的部分（宪法自检 + 155 Rust 测试全绿）

## 6. Consequences

### 正向后果
- 原批次 C（四步排版收敛）、批次 D（响应式底线）、动效语言、ScenePanel/promptStore 拆分合并为单一北极星方案（见 [[ui-reshape]]）
- 暗 band 失效等"旧账待复验"项在重塑中一并闭环

### 反向后果
- v0.1.0 延期；期间 main 分支冻结功能性新增（重塑走 `reshape/ui-v2` 分支）
- design-spec §9 角色表扩充（ph-action / ph-page-title / ph-num）随重塑回流，人审滞后于代码

### 未来反悔成本
- **代码改造规模**：前端 CSS 全量 + ScenePanel/promptStore 结构，约 30 文件；回退 = revert 分支合并 commit
- **数据迁移**：无（数据层不动）
- **不可逆点**：typography preset 迁移后旧 `font:` 简写基线不可自动还原（git 可回退）
