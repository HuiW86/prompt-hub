## 1. 项目定位

### 1.1 一句话定位

**一款服务于"AI 编程手动挡阶段"的桌面应用，主形态为快捷键唤起的全屏仪表盘、辅形态为副屏常驻视图，承载提示词资产的展示、调用和沉淀，以及人机协作的协议对齐。**

### 1.2 这不是什么

明确边界比描述功能更重要。这个项目**不是**：

- **不是**"更漂亮的提示词管理器"——市面上这类 App 已经很多（Snippety、Paste、Promptler），我们不和它们竞争
- **不是**"更快的提示词调用工具"——Raycast / 文本替换在调用速度上永远更快
- **不是**"AI 自动驾驶平台"——不涉及自动化执行、Agent 编排、workflow 引擎
- **不是**"面向团队协作的知识库"——单人使用，不考虑权限、评论、分享

### 1.3 这是什么

这个项目**是**：

- 一个**手动挡阶段**的练车场，帮使用者在高强度 AI 编程中建立稳定的认知节奏
- 一个**外部化的思考工具**，把"我可以怎么想这件事"从头脑里卸载到屏幕上
- 一个**资产沉淀场**，让每天的提示词使用自动产生可以复用、可以迭代的资产
- 一个**人机协议握手台**，让每次协作开始前的"对齐动作"显式化、工具化
- 一个**双形态承载**的仪表盘——主形态为快捷键唤起的全屏覆盖窗口（80% 使用场景），辅形态为副屏常驻视图（深度工作和复盘场景）

### 1.4 用户画像

**使用者是一人公司/独立开发者，高频使用 Claude Code 进行 AI 辅助编程。** 具备以下特征：

- 每天与 AI 对话频次 > 50 次
- 已经形成一套个人的提示词 / 工作流话术（几十条以上规模）
- 有持续迭代和优化自己工作方法的习惯
- 重视"看见全局"胜过"操作最快"
- 对 AI 编程范式有自己的演进路径规划（SOP 化 → 手动 → 自动 → 协同 → 自主）

## 2. 设计哲学

这九条哲学是整个项目的地基。**任何具体设计决策都必须能回溯到这里某一条**，否则这个设计就不该存在。

> 第 1-8 条原为 PRD §2.1-§2.8；第 9 条原为 PRD §4.6「迭代原则」，因属于元原则，拆分时并入哲学层。

### 2.1 哲学一：五阶段演进路径

AI 编程的成熟度分五个阶段，**这个项目只服务于第二阶段**：

| 阶段 | 特征 | 对应工具形态 |
|------|------|------------|
| 1. SOP 化 | 把隐性工作方法显式化为话术、模板 | 笔记本、文档 |
| **2. 手动挡** | **人完整驱动，工具辅助展示和调用** | **本项目** |
| 3. 自动挡 | 工具自动跑一段，人设定起点和终点 | Claude Code slash commands |
| 4. 辅助驾驶 | 人机实时协同决策 | Claude Code plan mode + subagents |
| 5. 自主驾驶 | AI 端到端交付 | 目前不可达 |

#### 为什么用"汽车驾驶"做比喻

选择汽车而不是常见的"初级/中级/高级"分级，不是修辞偏好，是因为汽车比喻**内嵌了两个其他分级不具备的核心前提**：

**前提一：驾驶员永远是决策者**。哪怕是最先进的辅助驾驶，方向盘依然在人手里——这和 AI 编程的理想状态完全一致。我们不是在训练 AI 替代自己，是在训练自己和 AI 形成稳定的协同关系。"初级→高级"这种分级暗示了"终点是人不再需要干预"，但汽车比喻明确拒绝这个暗示。

**前提二：每一阶段都需要前一阶段的积累**。不能跳过手动挡直接开辅助驾驶——你的脚会不知道该放哪里、你的眼睛会不知道该看哪里、你的肌肉会不知道该在哪一刻紧张。**能力的跃迁不是软件升级，是肌肉记忆的沉淀**。这一点对本项目极其关键：我们不是在做"替代思考的工具"，是在做"帮思考沉淀为习惯的工具"。

#### 每一阶段的关键转折点

从一个阶段到下一个阶段，**真正的转折点不是"技能熟练"，而是一次认知范式的切换**：

- **SOP 化 → 手动挡的转折**：当使用者意识到"我的工作方法不能只在脑子里了，需要外化成可调用的资产"。这一步的前提是使用者已经有稳定的工作方式可以被 SOP 化——新手直接搭手动挡工具会失败，因为他们的工作方式本身还没成型。

- **手动挡 → 自动挡的转折**：当某个 SOP 被重复执行过足够多次（建议 20 次以上），每次的路径几乎一致、分支判断稳定、结果可预测——这时候才值得自动化。**过早自动化会把不稳定的 SOP 固化成错误**，而修正成本远超手动的麻烦。

- **自动挡 → 辅助驾驶的转折**：当自动化 SOP 开始遇到"需要临场判断"的决策点，而且这些决策点频繁出现。这时候不是让 AI 自己决策（那是下一步），而是让人机能够实时交换信息、共同决策。

- **辅助驾驶 → 自主驾驶的转折**：这一步当前不可达，原因前面讨论过——决策点累积偏差让"全自主"在数学上就是 1% 以下的成功率。硬件性能和模型能力的提升都解决不了这个问题，只能等到 AI 能稳定地"在不确定时主动停下来问人"的那一天。

#### 跳级的风险

这是这个比喻最有价值的部分——**它明确告诉你跳级会在哪里翻车**：

- **跳过 SOP 化，直接搭手动挡工具**：会得到一个"看起来很厉害但每天都要重新想怎么用"的工具，三个月后弃用
- **跳过手动挡，直接配自动化**：自动化出来的结果漂浮在使用者真实工作流之外，每次都要手动改、还不如不自动化
- **跳过自动挡，直接上辅助驾驶**：人机协同的前提是双方都知道自己在做什么，使用者如果还没通过手动挡和自动挡建立起对 Claude 能力边界的真实感知，所谓"协同"只是单方面被 AI 牵着走
- **幻想自主驾驶**：把"AI 全包一个任务"当作目标，最终产物会系统性地偏离真实意图，而且偏离的地方往往是你自己最不擅长发现的地方

#### 本项目在五阶段中的精确位置

本项目**只做手动挡，不做相邻阶段**：

- **不做 SOP 化**：话术的创作、提炼、文字打磨应该在 Obsidian 或笔记软件里完成。本项目不是编辑器，是展示器和调用器。
- **不做自动挡**：一键触发自动化、编排多步执行、自主跑 agentic loop——这些应该在 Claude Code 的 slash commands、composite commands、subagents 机制里实现。本项目可以**为未来迁移到自动挡做准备**（比如 SOP 模板可以直接映射为 composite commands），但自身不执行任何自动化动作。

边界守住了，这个项目才能做深。边界一破，它就会变成另一个"想包打一切"的失败工具。

**设计上的含义**：

- 不追求 Raycast 式的"闭眼换挡"——那是自动挡的事
- 不追求 Workflow 引擎式的自动编排——那是辅助驾驶的事
- **核心是让使用者清楚地看到"我在做什么、我可以怎么做、下一步通常做什么"**

### 2.2 哲学二：看见全局 > 操作最快

对手动挡学员来说，**视觉全景比操作速度更重要**。

这条颠覆了效率工具的常见假设——多数效率工具追求"最少点击""最少步骤"。本项目主动接受"多一步点击"的代价，换取"一眼看全"的收益。

**设计上的含义**：

- 默认视图是全景式的，不是搜索框式的
- 相位、场景、话术、最近使用、SOP 进度应同屏可见
- 拒绝把关键信息隐藏在 Tab、下拉菜单、二级页面后面

### 2.3 哲学三：注意力分层通过"分离"实现，但分离方式因形态而异

**仪表盘和主工作屏必须存在某种"分离"，分离才能创造注意力分层。但分离的实现方式因形态而异——主形态用时间分离，辅形态用空间分离。**

认知科学上，把工具信息从主工作流卸载到一个"另外的地方"，形成 "external cognition offloading"（外部认知卸载）。主屏是执行现场，仪表盘是思考工具。**从执行态切回决策态需要一个触发动作**——这个动作不是为了换屏幕看，是为了换大脑模式。

**两种分离机制**：

| 分离方式 | 实现形态 | 触发动作 | 退出动作 |
|---------|---------|---------|---------|
| **时间分离**（主形态） | 快捷键唤起的全屏覆盖窗口 | 按下全局快捷键（如 ⌥ Space） | 复制后自动隐藏 / ESC 关闭 |
| **空间分离**（辅形态） | 副屏常驻视图 | 抬头看副屏 | 不需要退出（常驻在场） |

**为什么主形态选时间分离**：

- **不占视觉空间**——主屏 100% 留给和 AI 的对话，注意力不被持续干扰
- **召之即来**——任何笔记本/任何设备都能用，不需要副屏硬件
- **用完即走**——复制后窗口立刻消失，主屏粘贴流程顺滑
- **认知触发更强**——"按下快捷键"这个动作本身就是一次明显的模式切换，比"抬头看副屏"更显式

**为什么辅形态保留空间分离**：

- **持续在场**——状态指示（当前相位、SOP 进度）可被余光感知
- **深度工作场景**——做月度 review、整理资产时，需要工具长期可见
- **不依赖快捷键**——可以纯靠点击操作，适合非键盘党场景或思考时的随意浏览

**设计上的含义**：

- 项目形态**必须是桌面原生应用**（Tauri/Electron），不是浏览器 tab，不是浮层
- 同一个应用提供两种唤起方式：快捷键全屏（主） / 副屏窗口（辅）
- 两种形态共享同一套数据、同一套 UI 资产——只是承载介质和触发方式不同
- 详见 [[product-spec#4.0-双形态架构]]

**反方案**（明确拒绝的）：

- ❌ **主屏浮层**——既不时间分离也不空间分离，遮挡主屏破坏专注
- ❌ **浏览器 tab**——需要切换 tab 才能看，离手成本高且容易被关掉
- ❌ **只做副屏不做快捷键**——使用者 80% 时间没有/不开副屏，工具就废了
- ❌ **只做快捷键不做副屏**——失去"持续在场的状态指示器"价值

### 2.4 哲学四：使用即沉淀

每一次点击、每一次复制、每一次组合，**都应该成为资产的一部分**，而不是挥发掉的操作。

传统工具把"沉淀"当作一个**独立动作**（手动保存、手动命名、手动分类）——这个摩擦让 90% 的有价值信息都没被沉淀下来。

本项目的设计目标：**让沉淀成为使用的副产品**。

**设计上的含义**：

- 每次调用自动记录 usage_count、last_used_at
- 重复使用的组合自动识别并提示保存
- 使用热度数据驱动月度 review，而不是靠回忆

### 2.5 哲学五：话术是分层语言系统

经过观察，一个成熟的提示词库不是**扁平的列表**，而是**分层的语言系统**：

```
Modifier（原子）
    ↓ 组合
Composition（组合）
    ↓ 固化
Macro（宏定义 / Favorites）
```

- **Modifier（原子）**：不可再分的方法论动作，比如"第一性原理思考""启动子代理并行"
- **Composition（组合）**：由多个 Modifier 按顺序拼接而成的完整提示词
- **Macro（宏）**：高频使用的 Composition 被固化为单独的快捷入口

**设计上的含义**：

- 三层结构必须在 UI 上都有对应位置，不能只体现其中一层
- 三层之间的跃迁路径必须被设计（Composition → Macro 的固化、Modifier → Composition 的组装）
- 数据模型也必须支持这种跃迁的可追溯性（一个 Macro 由哪些 Modifier 组成？）

### 2.6 哲学六：减少"离手"但不消除

"离手"指的是：注意力从主屏（和 AI 对话）离开，去操作辅助工具。

**完全消除离手** = Raycast 式浮层 → 属于自动挡阶段
**完全不管离手** = 频繁切屏、手忙脚乱 → 过度原始

**手动挡阶段的正确选择**：把离手动作**显式化、标准化、最小化**。

- 显式化：离手是被设计的，不是意外发生的
- 标准化：离手的节奏固定（每次都是"抬头看副屏 → 选 → 回到 Mac"）
- 最小化：离手后的操作控制在 2 步以内（选场景 + 点话术）

这种受控的离手，反而创造了**思考的刹车片**——点击前那一瞬间的再确认，让误用率下降。

### 2.7 哲学七：人机协同需要协议对齐

和 AI 高效协作的前提，不是"我有多好的任务提示词"，而是**AI 知道我这次协作的模式是什么、要产出什么形态的东西**。一次清晰的相位对齐，胜过十句精雕细琢的任务话术。

**对齐话术定义"框"，任务话术在框里工作**。框不对，任务话术再精都无效；框一旦定下来，后续沟通摩擦大幅下降，输出形态可预测。

因此对齐话术是**协作的元层**——它不参与任务话术的组合、不被 SOP 引用、不和 Macro 互相转化。它独立成对象（AlignmentPhrase），独立成 UI 区域（相位带），独立成认知动作。

**设计上的含义**：

- AlignmentPhrase 是独立资产对象，不寄生在 Macro 里（详见 §3.5）
- 仪表盘上对齐话术有专属位置——相位带（详见 [[product-spec#5.1-相位带]]），优先级高于任何任务话术
- 协议层和内容层在数据、UI、使用流程上物理分离（详见 §8.6 边界声明）
- 相位切换本身是一种被珍视的"思维转场"，UI 应让这种转场可见

### 2.8 哲学八：双形态承载，按场景分工

仪表盘不存在"唯一正确的承载形态"——使用者一天的工作里，**有些瞬间需要召之即来的工具，有些瞬间需要持续在场的状态指示**。强行用一种形态覆盖两种需求，结果是两边都不顺手。

因此本项目采用**双形态承载**：

| 维度 | 主形态：快捷键唤起 | 辅形态：副屏常驻 |
|------|-----------------|---------------|
| **使用占比** | 80%（日常高频） | 20%（深度场景） |
| **触发方式** | 全局快捷键 | 永远在场 |
| **退出方式** | 复制后自动隐藏 / ESC | 不退出 |
| **核心价值** | 高效查找、快速调用 | 全景巡视、状态监视 |
| **典型场景** | 日常对话中复制话术 | 月度 review、整理资产、长任务跟随 |
| **优化方向** | 召之即来用完即走 | 持续可见余光感知 |
| **设备依赖** | 任何桌面（含笔记本单屏） | 需要副屏硬件 |

两种形态**共享同一套数据后端、同一套 UI 资产**——它们不是两个产品，是同一个产品的两种唤起方式。使用者切换形态的成本应该为零。

**设计上的含义**：

- 一套数据层服务两种形态（详见 [[prd#6-数据模型]]不动）
- 一套 UI 模块组件，两种容器形态（详见 [[product-spec#4.0-双形态架构]]）
- 使用者应清楚知道"现在该用哪一种"——这是手动挡阶段的元决策
- 实施时应**先做主形态 MVP**，再做辅形态（详见 [[plan#第一阶段]]）

**反方案**：

- ❌ 两种形态做成两个独立应用——数据同步会成为噩梦
- ❌ 主形态和辅形态 UI 完全不同——使用者要学习两套界面，认知负担过重
- ❌ 主形态只做"小浮层"——你的真实需求里主形态需要承载全部资产，不是 Raycast Snippets 那种轻量形态

### 2.9 哲学九：迭代原则——界面自身也是被维护的资产

> 原 PRD §4.6，因属于元原则（与"使用即沉淀"同源），拆分时上升为哲学第 9 条。

**真实仪表盘对应物**：赛车手会根据自己的习惯调整仪表盘布局——哪些信息放哪里，是可以被个人化配置的。

**决策**：界面本身必须可配置：

- Scene 的排序、显隐可调
- Phase 的增删、排序、改色可调
- Macro 的置顶数量可调
- SOP 模板可以自由增删
- Modifier 池可编辑、增删、分组重命名
- 首屏六个区域的高度比例可调

**理由**：

- 哲学四要求"使用即沉淀"——连使用者自己对界面的偏好也是一种沉淀
- 不同时期工作重心不同（这个月偏调研、下个月偏执行），界面应该能跟着变
- Phase 仍在演化（哲学七是新提出的实践），可配置是必需的

**反设计**：

- ❌ 把配置项藏在设置页（常用的配置应该在首屏就能调）
- ❌ 用复杂的设置系统（每个配置项都应该是"点一下就切换"的简单开关）

## 3. 核心概念

> 本章只保留概念定义和设计意图。**字段级定义、约束、Schema 见 [[prd#6-数据模型]]**。

### 3.1 三层资产模型

这是整个项目最核心的数据模型。其他所有设计都围绕这个模型展开。

#### Modifier（方法论原子）

**定义**：不可再分的方法论动作，是其他所有话术的"构造材料"。

**特征**：

- 通常以"动宾短语"形式存在，比如"主动调研、推进，""从更高的视角重新设计，"
- 末尾常带逗号而非句号——它是"半句话"，设计上就是要被拼接的
- 可按作用类型分组：认知前置（思维）、动作策略（执行）、产出形态（交付）、通用约束（约束）

**示例**：

- 认知前置："第一性原理思考""系统层面思考""信息不足先提问"
- 动作策略："主动调研推进""启动子代理并行""先调研对比方案"
- 产出形态："先出方案我拍板""制定实施计划""输出对比表格"
- 通用约束："确保不影响现有功能""用中文回答"

**设计要求**：

- 每个 Modifier 必须能独立成立，不依赖上下文
- Modifier 之间应尽量正交，避免语义重复
- 数量控制在 30 以内——过多会稀释组合的价值

**字段表**：→ [[prd#6.1-Modifier]]

#### Composition（组合）

**定义**：由多个 Modifier 按固定顺序拼接而成的完整提示词，服务于某个具体任务场景。

**标准结构**（推荐顺序）：

```
[角色设定] + [认知前置 Modifier] + [核心任务] + [动作策略 Modifier] + [产出形态 Modifier] + [通用约束 Modifier]
```

**示例**：

> 作为战略合伙人，（使用第一性原理思考，）全面负责本次方案的完善，（主动调研、推进，搜索最佳实践或 GitHub 开源实践作为参考，）减少实施路径，（输出经得起推敲的方案，先出方案供我拍板。）

**设计要求**：

- 拼接顺序固定，体现"先定认知 → 再选动作 → 再定产出 → 最后加约束"的节奏
- 拼接结果必须人类可读（不是机器 prompt，而是清晰的中文指令）
- 组合过程可追溯——任何 Composition 都能还原出使用了哪些 Modifier

**字段表**：→ [[prd#6.2-Composition]]

#### Macro（宏定义 / Favorites）

**定义**：高频使用的 Composition 被固化为单独的快捷入口，**绕过组合过程直接调用**。

**与 Composition 的区别**：

- Composition 是**现场组装**的（临时拼接）
- Macro 是**预先固化**的（一键调用）

**Macro 的来源**：

- 90% 应来自"曾经的 Composition"——使用者发现某个组合反复出现，就把它升级成 Macro
- 10% 可以是"原生的 Macro"——不由 Modifier 组合而来，而是直接写出来的成品话术

**示例**（典型 Macro）：

- "推翻重来"
- "借力最优解"
- "经得起推敲的方案"
- "第一责任人主动找事做"

**设计要求**：

- 每个 Macro 必须记录 `expandFrom`（由哪些 Modifier 组成），即使是原生写就的也标记为 `native: true`
- Macro 是整个系统里**使用频率最高的部分**，UI 上必须有最显眼的位置（"首屏 0 步抵达"）
- Macro 可以与 Composition / Modifier 形成闭环——Composition 沉淀为 Macro，Macro 也可以被"拆解"回 Modifier 去查看构成

**字段表**：→ [[prd#6.3-Macro]]

### 3.2 场景（Scene）

**定义**：按"使用时机"对话术进行的横切分类。

**示例**：规划（plan）、调研（research）、方案（debug）、排查、部署、审查、日常、工作流……

**与资产层级的关系**：Scene 是**正交维度**。一条 Modifier 可以属于某个 Scene，也可以跨 Scene 通用（大部分方法论 Modifier 其实是跨 Scene 的）。

**设计要求**：

- Scene 数量控制在 10 个以内——超过这个数就会变成"找场景"而不是"找话术"
- Scene 之间必须互斥——如果两个 Scene 里有高度重合的话术，要么合并要么重新切分
- Scene 可以有子状态（比如"方案"Scene 包含"生成→评审→修订→定版"四个子阶段），子状态在数据模型中以 `sub_stages[]` 字段承载

**字段表**：→ [[prd#6.4-Scene]]

### 3.3 SOP（标准作业流程）

**定义**：由多个 Macro 或 Composition 按**时间顺序**串联起来的工作流模板。

**与 Composition 的区别**：

- Composition 是**单次对话内**的提示词组合（横向拼接）
- SOP 是**跨多次对话**的话术使用序列（纵向串联）

**示例 SOP（方案设计型）**：

```
1. /借力最优解              → 让 Claude 先调研外部最佳实践
2. /全局视角出方案          → 基于调研结果出整体方案
3. /多 agent 评价方案       → 多 Agent 交叉评审
4. /根据 codex 意见修改     → 收敛修订建议
5. /更新方案                → 整合修订成最终版
```

**SOP 是这个项目第一个"超越单次对话"的概念**——它体现了使用者的工作流节奏，也是未来迁移到自动挡阶段的"半成品蓝图"。

**设计要求**：

- SOP 必须可视化呈现（流程图 / 带序号的步骤列表）
- 当前在 SOP 中的位置必须高亮标注（走到第 3 步时，第 3 步要突出）
- 每一步完成后，系统提示"下一步通常是什么"
- SOP 可以有分支（"如果评审通过则走 X，否则走 Y"），但不强制
- SOP **不**引用 Modifier（颗粒度不够）、**不**引用 AlignmentPhrase（哲学七边界，详见 §8.6）

**字段表**：→ [[prd#6.7-SOP]]

### 3.4 三个关键数据指标

所有指标都基于本地 localStorage 记录，不涉及后端。

- **usage_count**：每条话术/Modifier/Macro 被复制的次数，永久累加
- **last_used_at**：最近一次被复制的时间戳
- **session_recent**：本次会话中最近 5 次复制记录（用于"刚刚操作了什么"）

这些指标不是锦上添花，是**驱动哲学四（使用即沉淀）落地的核心机制**。没有它们，这个项目就退化回普通的"提示词陈列馆"。

**字段表**：→ [[prd#6.8-UsageRecord]]

### 3.5 认知相位（Phase）与对齐话术（AlignmentPhrase）

> 本节是 v0.3 新增的核心概念，对应哲学七。

#### 概念出处

使用者在长期 AI 编程实践中观察到：一次完整的人机协作任务，无论 Scene 是什么，内部都会经历一组**认知相位**——发散、理解、规划、生成、执行、收敛、沉淀、迭代。每个相位对 AI 的期待完全不同（比如"发散"期待铺开可能性，"收敛"期待给出结论）。

如果不在相位切换时主动告知 AI，AI 会按它默认的"任务执行模式"工作——这会导致大量沟通摩擦：你想要发散，AI 给你结论；你想要收敛，AI 还在铺细节。

因此使用者发展出一套实践：**在任务开始前、或对话中途切换相位时，先发一句"对齐话术"告诉 AI 当前的相位**。这套实践已经被验证有效，是本项目的核心特性。

#### Phase（认知相位）

**定义**：一组协作模式的分类容器。每个 Phase 代表一种特定的人机协同模式。

**当前实践版本**（可配置增删）：

| 相位 | 协同期待 |
|------|---------|
| 发散 | 铺开可能性，不下结论 |
| 理解 | 拆解问题、找出本质 |
| 规划 | 制定路径和优先级 |
| 生成 | 产出具体方案/代码/内容 |
| 执行 | 按既定方案推进，少质疑多动手 |
| 收敛 | 对比、评估、给出结论 |
| 沉淀 | 复盘、提炼、记录 |
| 迭代 | 基于反馈修订，推翻重来 |

**特征**：

- Phase 是**轻量级容器**，只服务于 AlignmentPhrase，**不污染**任何其他资产
- Phase 列表可配置增删（使用者仍在演化中，不写死成枚举）
- 一次完整的任务会经历多个 Phase 的切换（但不一定按上述顺序）

**字段表**：→ [[prd#6.5-Phase]]

#### AlignmentPhrase（对齐话术）

**定义**：人机协作的协议层话术。在任务开始前、或对话中途切换相位时使用，告诉 AI 当前的认知相位，定义本次协作的"框"。

**形态**：一句（或几句）完整话术。例如：

- 发散相位："我们现在做发散，先把所有可能性铺出来，不要给结论，不要评估优劣。"
- 收敛相位："我们现在做收敛，从前面的方案里选出最优解，给出明确推荐和理由。"
- 沉淀相位："我们现在做复盘，目标是提炼可复用的方法论，不是重新评估方案本身。"

**特征**：

- **独立对象**——不寄生在 Macro 里，独立成 AlignmentPhrase 类型
- **属于 Phase**——每条 AlignmentPhrase 必须归属于一个 Phase
- **孤岛声明**——不参与 Modifier 拼接、不属于任何 Scene、不被 SOP 引用、不和 Macro 互相转化（详见 §8.6）

**为什么独立成对象**：

对齐话术决定了**这次协作的方式和最终输出形态**——它是"协议层"。Macro 是"任务层"。两者混在一起会让使用者失去"先定框、再发任务"这个核心认知节奏。哲学七的硬边界要求两者在数据和 UI 上物理分离。

**字段表**：→ [[prd#6.6-AlignmentPhrase]]

#### Phase 和 Modifier 四组的关系

PRD 中 Modifier 按四组分类（认知前置/动作策略/产出形态/通用约束）。Phase 看起来和这四组类似，但**两者是不同维度，并存不冲突**：

- **Modifier 四组** = 拼接顺序中的语法位置（这个 Modifier 在句子里放在哪一段）
- **Phase** = 认知动作的相位归属（这条对齐话术服务于哪个思维相位）

Modifier 四组固定不变（枚举），Phase 可配置增删（独立表）。两者不交叉关联——Modifier 不带 Phase 标签，AlignmentPhrase 不参与 Modifier 拼接。

## 8. 不做什么（Non-Goals）

这一章比"做什么"更重要。**明确拒绝的事情，决定了这个产品的性格**。

### 8.1 不做 AI 生成

- 不内嵌任何 "用 AI 帮我生成 Modifier" 的功能
- 不内嵌 "AI 自动分类我的话术" 的功能
- 不做 "AI 自动建议 SOP" 的功能
- 不做 "AI 自动判断当前相位" 的功能

**理由**：

- 引入 AI 生成会让使用者失去"自己思考"这一步，而这一步恰恰是手动挡阶段的核心价值
- 本工具的目的是沉淀**使用者自己**的方法论，不是让使用者消费 AI 生成的通用话术
- 这个功能一旦加入，边界会无限扩大（用户会期待 "AI 帮我优化现有 Modifier" 等），产品会走形
- 特别地，相位识别必须由使用者主动完成——这是哲学七要求的"协议握手"动作的核心价值

### 8.2 不做社区/分享功能

- 不做"导出为可分享的链接"
- 不做"浏览他人的提示词库"
- 不做"团队协作"

**理由**：

- 单人工具的价值最大化依赖**高度个性化**——社区化会让设计向"通用好用"妥协
- 使用者的话术沉淀涉及其工作方式的指纹——分享机制反而产生隐私压力
- 想看别人怎么做的，去 GitHub 看别人的 CLAUDE.md 就行，不需要在这个工具里做

### 8.3 不做自动执行

- 点击话术只复制到剪贴板，不自动发送给任何 AI
- 不集成 Claude API 直接对话
- 不做"一键跑完整个 SOP"的自动化
- 不做"切换相位时自动发对齐话术给 Claude"——必须人主动粘贴

**理由**：

- 哲学一明确本项目只服务于手动挡阶段——自动执行属于自动挡
- 自动执行会隐藏中间过程，剥夺使用者观察和调整的机会
- 手动复制粘贴虽然多一步，但这一步是**思考的缓冲**，有认知价值

### 8.4 不做多层级文件夹

- Scene 下不允许再嵌套子 Scene（SubStage 是同级分组，不是嵌套）
- Macro 不允许分组到文件夹
- Modifier 只有四个固定分组，不允许自定义分组层级
- Phase 下不允许再嵌套子 Phase

**理由**：

- 层级 > 1 就会变成"找东西"——违背哲学二（看见全局）
- 复杂的层级会给未来的月度 review 带来巨大负担
- 扁平 + tag 是更好的组织方式（但本项目当前甚至不做 tag，用 Scene + 四组 Modifier + Phase 已经够分）

### 8.5 不做付费/商业化

- 这是自用工具，不考虑商业化路径
- 不需要用户系统、订阅机制、数据加密传输等商业化必需的复杂性

**理由**：目标是沉淀使用者自己的方法论，商业化会扭曲设计方向。

### 8.6 不让对齐层与任务层混淆

> 本节是 v0.3 新增边界，承接哲学七。

**协议层（AlignmentPhrase / Phase）和任务层（Modifier / Composition / Macro / Phrase / Scene / SOP）必须物理分离**：

- AlignmentPhrase 不出现在 Composition 工作台（不参与 Modifier 拼接）
- AlignmentPhrase 不能被加入 SOP 步骤
- AlignmentPhrase 不与 Macro 互相转化（即使内容相似也是两个对象）
- Macro 区不展示 AlignmentPhrase
- 相位带（Phase Bar）不展示任何任务话术
- Scene 不包含 AlignmentPhrase

**理由**：

- 哲学七要求"框"和"内容"分离。一旦在 UI 或数据上混淆，使用者就会失去"先定框、再发任务"这个核心认知节奏
- 数据上混淆会让"相位使用分布"等指标失真——分不清是任务话术还是协议话术
- UI 上混淆会让使用者在视觉上不再能瞬间区分"这是要切换协作模式"还是"这是要发任务"

### 8.7 不让主辅形态各自走偏

> 本节是 v0.5 新增边界，承接哲学八。

**主形态和辅形态共享同一套数据后端和 UI 模块——它们不是两个产品**：

- 不做"主形态独占功能"和"辅形态独占功能"的对立——所有核心功能两种形态都能完成
- 不做两套独立 UI——主辅共用 UI 组件，只是容器和触发方式不同
- 不做数据同步——两种形态读写同一份本地数据，零同步层
- 不做"形态偏好"配置——使用者只是在不同时刻选择不同入口，不需要预先声明
- 不做主形态和辅形态之间的数据迁移——它们就是同一份数据

**允许的差异**（不算"走偏"）：

- 主形态有"复制即隐藏"，辅形态没有
- 主形态背景半透明，辅形态不透明
- 辅形态可以把 Composition 工作台作为常驻侧栏（深度场景需要）
- 辅形态的状态仪表会展示更长周期的数据趋势（因为可持续查看）

**理由**：

- 哲学八要求"双形态承载，按场景分工"——分工不是分裂
- 数据分裂的代价是"使用者不知道自己刚才的沉淀去哪儿了"——这是最致命的反沉淀
- UI 分裂的代价是"两套界面学习成本翻倍"——这是最致命的反扫视

## 10. 开放问题

这些是本 PRD 当前无法确定、需要在实施过程中根据真实使用反馈决策的问题。

### 10.1 Modifier 池的增长管理

- 什么时候该新增 Modifier？什么时候应该修改已有的 Modifier 而不是新增？
- Modifier 数量超过 30 条时，是应该拆分为多组，还是应该合并同类项？
- 是否需要引入 Modifier 的"等级"（核心、常用、冷门）？

**观察期**：使用 3 个月后，根据 usage_count 分布 + UsageRecord.modifier_ids 的组合频次决定。

### 10.2 SOP 的粒度

- SOP 步骤应该是"一次复制操作"还是"一个对话阶段"？
- 如果某个步骤需要多轮对话才能完成（比如"方案评审"可能涉及多次来回），SOP 怎么表达？
- SOP 中的分支逻辑（if-else）是否值得支持？

**观察期**：第三阶段实施后，根据实际使用 SOP 的情况 + UsageRecord.sop_id / sop_step_order 数据决定。

### 10.3 手动挡 → 自动挡的过渡点

- 什么时候应该把某个 SOP 从"手动执行"升级为 Claude Code 的 composite command？
- 升级之后，仪表盘里的 SOP 模板应该保留吗？（作为可视化文档？还是删除？）
- 如何避免"升级太早"——某个 SOP 在手动阶段还没被验证稳定就被自动化？

**观察期**：有至少 2 个 SOP 被使用超过 20 次之后，讨论迁移。

### 10.4 Phase 体系的稳定性

> v0.3 新增开放问题。

- 当前 8 个 Phase 是否稳定？会不会演化成 10 个或缩减为 6 个？
- 是否会出现"复合相位"（同时具有发散和收敛的特征）？
- 每个 Phase 下的 AlignmentPhrase 应该有多少条才是合适的？1 条够吗？还是按场景分多条？
- 是否需要"相位转场话术"——专门服务于"从 A 相位切换到 B 相位"的过渡话术？

**观察期**：MVP 使用 1-2 个月后，根据 UsageRecord 中 phase_id 的分布和 AlignmentPhrase 的实际使用情况决定。

### 10.5 多人使用的可能性

- 如果未来扩展到团队（比如带一个助理），数据模型怎么扩展？
- 是否需要"共享 Macro 池 + 个人 Macro 池"的双层结构？

**当前决策**：**不考虑**。按 8.2 原则坚决拒绝多人功能。未来若真需要，另起项目。

### 10.6 Scene 区在 user flow 中的存在感

> v0.4 新增开放问题。

[[product-spec#4.5.4-跨场景对照表]] 显示：三个典型场景（SOP驱动、单点调用、现场组装）中，**Scene 全景区都没有作为主角出场**。这是一个需要关注的信号：

- 是否 Scene 这个维度对"任务流程"的指导价值不如对"资产管理"的价值？
- Scene 区可能更多承担"事后回看 / 整理 / 浏览"的角色，而非"任务进行中"的角色？
- 如果是这样，Scene 区在首屏的视觉权重是否应该下调？

**观察期**：MVP 使用 1-2 个月后，根据 UsageRecord.source 的分布（来自 'scene' 的占比有多少）决定是否调整 Scene 区的设计定位。

**当前决策**：**暂不调整**。Scene 区仍按 PRD 原设计实现，保留首屏位置——因为"看见全局"哲学要求即使低频使用的资产容器也必须可见。

### 10.7 搜索语法演化

> v0.5 新增开放问题。

v0.5 的搜索区（[[product-spec#5.0-搜索区]]）只做全文模糊匹配，不支持任何过滤前缀或语法。未来是否需要扩展取决于实际使用反馈：

- 当资产规模超过 100 条时，全文搜索的结果会不会过多？
- 是否需要 `#macro` `#scene:方案` `#phase:发散` 这种过滤前缀？
- 是否需要支持时间过滤（"上周用过的"）？
- 是否需要 `>` 命令模式（如 `> new macro` 直接创建）？

**观察期**：MVP 使用 2-3 个月后，根据搜索的实际使用频次和"搜不到时的痛苦程度"判断。

**当前决策**：**先不做语法**。哲学二要求扫视为主，搜索是兜底——如果需要复杂语法才能找到东西，说明扫视已经失效，那是更严重的问题，要先解决资产组织而非搜索功能。

### 10.8 双形态的使用比例验证

> v0.5 新增开放问题。

哲学八假设主形态承载 80%、辅形态承载 20%。这是基于使用者当前的工作模式推断，但未经长期验证。

可能的情况：

- **比例符合**：主辅 80/20，PRD 的所有节奏设计有效
- **主形态更重**：实际可能 95/5——则辅形态可以简化为"只有月度 review 一个用途"
- **辅形态更重**：实际可能 50/50——则需要重新平衡两种形态的设计深度
- **形态使用与场景强绑定**：可能某些 Scene 严重偏向主形态，某些严重偏向辅形态——这会影响 [[product-spec]] 各模块的优化方向

**观察期**：辅形态实现后（第五阶段）至少使用 1 个月，根据 UsageRecord 数据判断两种形态的实际使用占比。

**当前决策**：**按 80/20 假设推进**。主形态先做，辅形态后做，第五阶段实施前根据观察数据调整辅形态的设计深度。

## 11. 附录：核心哲学回顾

为方便快速回顾，这里汇总九条核心哲学：

1. **五阶段演进路径**——本项目只服务于手动挡阶段
2. **看见全局 > 操作最快**——视觉全景比点击速度更重要
3. **注意力分层通过"分离"实现，但分离方式因形态而异**——主形态时间分离，辅形态空间分离
4. **使用即沉淀**——沉淀是使用的副产品，不是独立动作
5. **话术是分层语言系统**——Modifier → Composition → Macro 三层结构
6. **减少"离手"但不消除**——保留思考的刹车片
7. **人机协同需要协议对齐**——对齐话术定义"框"，与任务话术物理分离
8. **双形态承载，按场景分工**——主形态快捷键唤起（80%），辅形态副屏常驻（20%），共享数据与 UI
9. **界面自身也是被维护的资产**——可配置即沉淀（v0.5 拆分时由原 §4.6 升格为哲学）

**任何未来的功能讨论，都应该先问：这符合哪一条哲学？如果九条都不沾，这个功能就不该加。**
