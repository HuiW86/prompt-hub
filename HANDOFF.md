## Objective

Plan (not yet build) two capabilities for the Dashboard:
1. **Non-rigid layout** — regions size to content (few Macros ≠ big empty
   block) + user-draggable dividers with persisted ratios.
2. **Full editing** — create / delete / rename / edit-content / drag-sort
   across all 4 asset types (Modifier / Composition / Macro / AlignmentPhrase).

Discipline: research best practices first, design before code, open ADR
before pulling new deps.

  Macro/AlignmentPhrase. No reorder commands.

- **Key research findings folded into plan:**
  - Lock `@dnd-kit/react@0.4.0` + `@dnd-kit/helpers@0.4.0` (new API: no
    `SortableContext`/`arrayMove`; use `DragDropProvider` + `move` +
    `isSortable`/`initialIndex`; React 19 must use this, NOT legacy core/sortable).
  - `react-resizable-panels` **v4** chosen (omar approved): renamed API
    `Group`/`Panel`/`Separator` + `useDefaultLayout`; **percent-only, no px**.
  - Design: centralized edit entry (Raycast/Linear), drag as keyboard-augment
    layer, persist layout ratio (Obsidian lesson), delete tiering (soft→toast /
    hard→confirm). Avoid Notion multi-modal anti-pattern.
  - **NSPanel key-window (ADR-014) is a prerequisite** for dnd-kit keyboard
    sensor + resize-handle focus — must verify in NSPanel (new task P1.5).
- **Plan written + amended to v0.2 + committed (`1cdcafb`):**
  `docs/plans/asset-editing-and-adaptive-layout.md` — 4 phases (P1 Macro full
  chain → P2 other 3 assets → P3 content-adaptive → P4 resizable dividers),
  each with impact / quantified acceptance / risks. `status: pre-code`.

## Risks & Decisions

- **ADR-016 gates P1.4** — if review overturns dnd-kit, P1.4 reworks. P1.0 is a
  hard blocker (no deps before Accepted).
- **react-resizable-panels v4 = percent-only** — token system is px → need
  px↔% conversion util; re-compute on window resize (stale otherwise).
- **NSPanel focus** — keyboard sensor + resize handle depend on key-window
  (ADR-014 `canBecomeKeyWindow`); no public lib×NSPanel conflict found but must
  verify live (P1.5).

  render source + `isDragging` ref gate + cancel/fail snapshot rollback.
- **B2 physical separation** — AlignmentPhrase edit UI must not leak into
  Composition/Macro areas (P2.5 self-check, 3-item checklist).
- **Modifier/Composition read gap** — no front-end `list_*` for these today;
  P2.2 must探明 surfacing location before building their edit UI.
