## Objective

## Completed (this session)

- **Root cause (codex + live-instrumented, triple-confirmed):** the borderless
  overlay never became the macOS **key window**; AppKit delivers keyboard events
  only to the key window. The existing isa-swizzle to NSPanel was insufficient ‚Äî
  a borderless window (no Titled style bit) returns NO from the default
  `canBecomeKeyWindow` even as an NSPanel, so `makeKeyWindow()` was a no-op.
  Live probe proved it: `class="NSPanel" canBecomeKey=false` ‚Üí after fix
  `class="PromptHubKeyablePanel" canBecomeKey=true isKey=true`.
- **Fix in `src-tauri/src/macos.rs`:**
  - Added `define_class!` subclass `PromptHubKeyablePanel : NSPanel` overriding
    `canBecomeKeyWindow` / `canBecomeMainWindow` ‚Üí `true` (Raycast / tauri-nspanel
    pattern). isa-swizzle target switched from bare `NSPanel` to this subclass
    (no ivars ‚Üí instance_size unchanged, existing size assert still guards).
  - `setBecomesKeyOnlyIfNeeded(true)` ‚Üí `false`.
  - Added `make_key()` (`makeKeyWindow`) and `wake()` = order_front ‚Üí make_key ‚Üí
    focus_view, a single shared wake sequence.
- **`src-tauri/src/lib.rs` + `src-tauri/src/commands.rs`:** both wake paths
  (global-shortcut handler + `show_window` IPC) now call `macos::wake()` so the
  sequence can't drift between call sites.
- **`src/components/SearchBar.{tsx,module.css}`:** removed leaked `ÂÖúÂ∫ï`
  scaffolding text (phase-1 `a14d61e` residue) + its `.fallback` CSS class.
